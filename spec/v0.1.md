# Agent-Native Dev Conventions

Status: v0.1-beta — open for feedback

A numbered, prescriptive framework for structuring software projects so AI agents
can navigate them effectively from the first message — without extensive grepping,
without reading irrelevant files, without losing context.

Inspired by [The Twelve-Factor App](https://12factor.net) and
[Conventional Commits](https://www.conventionalcommits.org): spec-first, with a
canonical URL, tooling to follow.

---

## Background

Software engineering conventions were designed for human cognition: spatial memory,
hierarchical folder browsing, linear documentation reading. When a Claude Code agent
opens a Swift project with 40 files or a Next.js app with 30 pages, it navigates
blind — grepping, reading irrelevant files, losing context, making mistakes a
familiar human developer wouldn't make.

The problem isn't that agents are bad at navigating. It's that projects weren't
structured to be agent-navigable. No mainstream engineering convention addresses this.

`AGENTS.md` and `CLAUDE.md` (2024–2025) were the first formats with AI as primary
reader. 60k+ repos adopted them. But they're still written as dense READMEs by
humans for humans. The pattern exists; the optimization doesn't.

This framework defines what that optimization looks like.

## The Conventions

### I. Curated Context, Not Dense README

**The convention:** `CLAUDE.md` (or `AGENTS.md`) should contain only what grep and
glob cannot find. Before adding any line, ask: "Can an agent discover this by reading
files or running a search?" If yes, cut it.

**In practice:**

```markdown
<!-- Good: genuinely non-obvious facts -->

## Hot files
- `Sources/App/Services/AuthService.swift` — central auth actor; all user state flows
  through here; changes require updating AuthServiceTests
- `Sources/App/Models/User.swift` — schema changes here break 3 migration files

## Invariants
- Never store tokens in UserDefaults — use `KeychainHelper.swift`
- All new screens must be registered in `AppCoordinator.swift`
```

**Anti-pattern:**

```markdown
<!-- Bad: what grep can find -->

## Project overview
This is a Swift app that uses SwiftUI and MVVM architecture. It has a login screen,
a home screen, and a settings screen. The main entry point is AppDelegate.swift.
We use Combine for reactive bindings and follow clean architecture principles.
```

**Why it matters:** Short `CLAUDE.md` files with genuine non-obvious information
reduce agent runtime by ~28% and token usage by ~16% compared to dense READMEs
(arxiv 2601.20404). Narrative context wastes the context window agents need for code.

---

### II. Declare Hot Files Explicitly

**The convention:** List files that are central (many importers), volatile (change
frequently), or cross-cutting (affect multiple features). Do this in `CLAUDE.md`
or a dedicated index. Without this map, agents navigate blind.

**In practice:**

```markdown
<!-- TypeScript/Next.js example -->

## Hot files
- `src/lib/db/schema.ts` — central: changing this requires a migration; imported by
  15+ files across the codebase
- `src/middleware/auth.ts` — cross-cutting: all protected routes depend on this; touch
  it only after reading the auth flow in `.claude/docs/auth-flow.md`
- `src/components/Layout.tsx` — volatile: changes with every design iteration; not
  a good file to leave half-done between sessions
```

**Anti-pattern:**

```markdown
<!-- Bad: structure without significance -->

## Structure
The project follows standard Next.js conventions with pages/, components/, and lib/
directories. See the README for details.
```

**Why it matters:** Hot file declarations change the agent's navigation strategy for
the entire session. An agent that knows which files are architecturally significant
reads fewer irrelevant files and avoids making changes that have unintended cascades.
It's the highest-ROI sentence you can write in a project context file.

### III. Atomic Rules, Not Narrative Guidance

**The convention:** Each rule should encode exactly one verifiable invariant.
"Follow security best practices" cannot be acted on. "Never store tokens in
UserDefaults; use `KeychainHelper.swift`" can be checked in code review and violated
exactly once before the violation is visible.

**In practice:**

```markdown
<!-- Good: checkable invariants -->

## Rules
- Never call `URLSession` directly — all network requests go through `APIClient.swift`
- New database tables require a migration file in `db/migrations/` named
  `YYYY-MM-DD-description.sql`
- Feature flags live in `FeatureFlags.swift` — never hardcode `if user.isPremium`
  outside that file
```

**Anti-pattern:**

```markdown
<!-- Bad: narrative that requires interpretation -->

## Development guidelines
Follow security best practices and maintain code quality. Use appropriate abstractions
and follow the existing patterns in the codebase. Be consistent with how things are
done elsewhere in the project.
```

**Why it matters:** Narrative guidance is interpreted differently by each agent
session, each developer, and each code review. Atomic rules are either followed or
not — they're checkable, and violations are visible. They can eventually become
lint rules (automated enforcement is the natural next step).

### IV. Separate Spec from Rationale

**The convention:** Specification (what must be true) belongs in rule files, loaded
on every relevant task. Rationale (why this decision was made) belongs in decision
logs or inline comments, read once. Don't mix them.

**In practice:**

```markdown
<!-- .claude/rules/auth.md — spec, loaded on every auth task -->

## Auth rules
- Auth tokens live in Keychain, never UserDefaults
- Session expiry triggers re-auth flow via `AuthCoordinator.reauth()`
- OAuth callbacks are handled exclusively in `OAuthCallbackHandler.swift`
```

```markdown
<!-- docs/decisions/2024-09-auth-keychain.md — rationale, read on demand -->

# Decision: Keychain for token storage

After a security audit in September 2024, we evaluated UserDefaults vs Keychain.
UserDefaults is not encrypted on device. Given our enterprise customers with SOC 2
requirements, we accepted the implementation overhead. See the audit report at
[link]. Revisit if we drop enterprise tier.
```

**Anti-pattern:**

```markdown
<!-- Bad: rationale and spec mixed, bloating CLAUDE.md -->

We use Keychain for token storage. We evaluated UserDefaults but it's not encrypted;
after a security audit in Q3 we decided the risk wasn't acceptable given our user
base includes enterprise customers with SOC 2 requirements, so we use KeychainHelper
which wraps the Keychain API with error handling...
[50 more lines of rationale]
```

**Why it matters:** Rationale is needed once — to understand a past decision. Spec
is needed on every task. Mixing them bloats the hot-memory layer with context that
amortizes over time, not per-session. Separating them lets agents load the right
layer for the right purpose and keeps `CLAUDE.md` lean.

### V. Cold Memory for Deep Context

**The convention:** Detailed documentation — data models, UX flows, architecture
diagrams, API schemas — should live in separate files, referenced by path in
`CLAUDE.md`. Agents fetch on demand. Don't inline everything in the hot-memory layer.

**In practice:**

```markdown
<!-- CLAUDE.md (hot memory — always loaded) -->

## Deep context (read when needed)
- Data model: `.claude/docs/data-model.md`
- UX flows: `.claude/docs/ux-flows.md`
- API schema: `api/openapi.yaml`
- Architecture overview: `.claude/docs/architecture.md`
```

```markdown
<!-- .claude/docs/data-model.md (cold memory — loaded on demand) -->

## Data model

### User
| Field | Type | Notes |
|---|---|---|
| id | UUID | Primary key |
| email | String | Unique, indexed |
...
[100+ lines of schema detail]
```

**Anti-pattern:**

```markdown
<!-- Bad: schema inlined in CLAUDE.md -->

## Data model
User { id: UUID, email: String, createdAt: Date, updatedAt: Date, deletedAt: Date?,
roles: [Role], preferences: UserPreferences, ... }
Role { id: UUID, name: String, permissions: [Permission], ... }
[200 more lines of schema]
```

**Why it matters:** The context window is finite. Inlining large documents displaces
code and actual task context. The "Codified Context" architecture (arxiv 2602.20478),
validated across 283 sessions, uses exactly this pattern: a lean hot-memory
constitution (~660 lines) plus cold-memory documents fetched on demand via search.
Hot memory should be a map, not the territory.

### VI. Convention-Over-Description Naming

**The convention:** Name files, directories, and functions so their role is
self-evident without reading them. An agent reading `UserAuthService.swift` knows
it handles user authentication. `Helper.swift` and `Utils.ts` require investigation.

**In practice:**

```text
# Good — role is self-evident from the name

Sources/App/Services/UserAuthService.swift
Sources/App/Repositories/OrderRepository.swift
Sources/App/Coordinators/OnboardingCoordinator.swift

src/hooks/useCartState.ts
src/api/products/list.ts
src/lib/payments/stripe-webhook.ts
```

```text
# Bad — requires reading (or a CLAUDE.md entry) to understand

Sources/App/Services/Helper.swift     # what kind of help?
Sources/App/Misc/Manager.swift        # manages what?
Sources/App/Core/Processor.swift      # processes what?

src/utils/stuff.ts                    # what stuff?
src/lib/common.ts                     # common to what?
```

**Anti-pattern:** A file named `Manager.swift` that handles 7 unrelated concerns,
documented only in `CLAUDE.md`:

```markdown
# CLAUDE.md
Manager.swift handles auth, session lifecycle, feature flags, and remote config.
```

**Why it matters:** Every file an agent reads to understand its purpose costs tokens
and increases the chance of missing relevant context elsewhere. Convention-over-description
naming is the cheapest form of documentation — it requires no maintenance, can't go
stale, and works for both agents and humans. Files that do too many things should be
split, not explained.

### VII. Document Decay by Reference

**The convention:** Every documentation block that describes specific code should
reference the file it describes. When the referenced file changes substantially, the
documentation is suspect. Use `<!-- refers-to: path/to/file -->` or an equivalent
convention for your format.

**In practice:**

```markdown
<!-- refers-to: Sources/App/Services/AuthService.swift -->

## Auth flow

1. User submits credentials → `AuthService.authenticate(email:password:)`
2. On success: token stored via `KeychainHelper.save(token:for:)`
3. On failure: `AuthError` propagated to `AuthViewModel`, shown as inline alert
4. Session refresh: automatic, triggered 5 min before expiry
```

When `AuthService.swift` changes substantially, this annotation is the signal — for
the agent, for reviewers, and for future automated tools — that this doc may be stale.

**Anti-pattern:**

```markdown
<!-- Bad: no reference — doc rots silently -->

## Auth flow
1. User submits credentials → AuthService.authenticate()
2. On success: token stored in KeychainHelper
3. On failure: AuthError propagated to AuthViewModel
```

**Why it matters:** Documentation without a reference to the code it describes has
no staleness signal. Agents read outdated docs with full confidence, propagating
errors into new code. A `refers-to` annotation is the minimal viable decay mechanism.
It doesn't prevent rot, but it makes rot visible — to agents, to reviewers, and
eventually to tools that check git history against doc references.

## Adopting these conventions

These conventions are independent — adopt any subset. Start with the one that causes
the most friction in your current project.

**Highest ROI starting points:**

1. **II — Declare hot files** if your agent frequently touches the wrong files
2. **I — Curated context** if your `CLAUDE.md` is over 100 lines
3. **III — Atomic rules** if your agent ignores your style guidelines

Each convention builds on the others, but none requires the others to be useful.

**Reference implementations:** [`examples/`](../examples/) contains mock projects for
Swift/SPM and TypeScript/Next.js with all seven conventions applied and annotated.
